---
title: "SAMP1 The Wasteland"
---

Here is a harm reductionist approach to ebbing the flow of slop. Add a top level folder to the repo called, the_wasteland or wasteland or the ruins or something equally evocative.

In the wasteland:
    - Code can come from any source, AI human or both
    - There are no quality guarantees
    - There are no quality expectations
    - minimal docs, little concern for coherence minimalism or performance
    - A bot deletes files that have not been modified in a while
    - All code committed to the wasteland should be relatively easy to re-prompt into existence
    - No CI tests
    - Outputs are for internal use only, anything released from the wasteland must be tagged in some way to make this clear

Outside the wasteland:
    - On main, nothing can import code from the wasteland
    - Code must be read, line for line by a human author and at least one human reviewer
    - All code has a human owner who understands what the code does, how it does it and why it exists
    - There is an expectation of quality, the code is coherent, reasonable, documented, performant, minimal ect
    - Outputs can be released to customers

Will this slow down development?

Initially it might, but in the long term slow but correct is faster than fast but incorrect.
Mere activity should not be rewarded.

The wasteland 
- Prevents conflict / resentment via disclosure
- Protects a culture of trust / craft / quality 
- Reinforces a sense of code ownership / stewardship / liability

All while permitting the sugar high of rapid iteration, in a bounded way. Manic episodes can produce masterpieces!

If the wasteland becomes a surreal landscape where sense and nonsense meet, that's fine. Files that haven't been edited for a while eventually get deleted by the wasteland reaper. You can always revisit old gc'd ideas in source control. If a wasteland project grows into a stable dependency used by other wasteland projects, that's a good sign its time to port it to the human audited part of the repo. The person who does the translation owns the final output. They are the de facto authority on the code they ported, they should be able to answer questions about the code and yes if something goes wrong they may be held accountable. 

in some limited cases it might be ok for customers to interact with wasteland code, but only if they know what they are getting themselves into. The term "beta" is not strong enough.
